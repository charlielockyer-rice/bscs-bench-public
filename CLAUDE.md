# BSCS Bench - Agent Evaluation Framework

> **Fresh clone?** Run `bin/setup` for interactive setup, or see [docs/QUICKSTART.md](docs/QUICKSTART.md) for manual steps. Run `bin/doctor` to validate your environment.

## Project Overview

This project provides a **reusable agent evaluation framework** for testing AI agents on programming assignments. The framework is generic and can be adapted for any course.

**Included Courses:**
- `comp140/` - COMP 140: Computational Thinking (Python, 7 modules, 248 tests + written)
- `comp182/` - COMP 182: Algorithmic Thinking (Theoretical + hybrid, 9 homeworks, LLM-graded + Docker for HW4/HW6)
- `comp215/` - COMP 215: Principles of Software Development (Java, code-only)
- `comp310/` - COMP 310: Advanced OOP (Java, 6 homeworks + design, ~95 tests + LLM-graded, Docker-based)
- `comp321/` - COMP 321: Introduction to Computer Systems (C, 6 projects + written)
- `comp322/` - COMP 322: Fundamentals of Parallel Programming (Java/HJlib, 4 homeworks + written, Docker-based)
- `comp341/` - COMP 341: Practical Machine Learning (Python, 7 homeworks + written, sklearn + PyTorch)
- `comp382/` - COMP 382: Reasoning about Algorithms (Theoretical, 7 homeworks, LLM-graded)
- `comp411/` - COMP 411: Principles of Programming Languages (Java, 6 assignments, code-only)
- `comp421/` - COMP 421: Advanced Operating Systems (C + theoretical, 3 labs + 2 exams, SSH-based on CLEAR)

**Architecture:**
- `framework/` - Generic evaluation engine (Python courses)
- `agent/` - TypeScript agent harness with AI SDK 6 + OpenRouter

## Workspace Structure

Each workspace represents a **single task** (one assignment). Workspaces are self-contained with flat structure:

```
workspaces/
  circles/                     # Single-task workspace (one per assignment)
    workspace.yaml             # Metadata: assignment_number, assignment_name, course
    workspace.json             # JSON fallback for systems without PyYAML
    instructions.md            # Assignment instructions
    template.py                # Clean template (reference)
    solution.py                # Agent's working file
    writeup.md                 # Written responses
    attempts.json              # Grading attempt history
    lib/ -> symlink            # Course helper modules
```

**workspace.yaml:**
```yaml
assignment_number: 1
assignment_name: "circles"
display_name: "Circles"
course: "comp140"
language: "python"
assignment_mode: "hybrid"   # code | hybrid | written
created_at: "2024-01-20T10:30:00"
depends_on: "comp310_hw02"  # Optional: references a previous workspace
```

`assignment_mode` is generated by `bin/setup_workspace` and is the runtime
source of truth for whether an assignment is code-only, code+writeup, or
written-only. Prompting and grading should prefer this field over ad-hoc
filesystem heuristics.

### Sequential Dependencies (`depends_on`)

Some courses have sequential assignments where each builds on the previous one
(e.g., COMP 310 Ballworld HW01 -> HW02 -> HW03). The `depends_on` field in
workspace.yaml references the workspace name of the prerequisite assignment.

**How it works:**
1. Course config (e.g., `comp310/configs/hw3.yaml`) includes `depends_on: "comp310_hw02"`
2. `bin/setup_workspace` reads this and writes it to `workspace.yaml`/`workspace.json`
3. Before the agent runs (`bin/bench-cli`), dependency resolution kicks in:
   - Finds the referenced workspace in `workspaces/`
   - Identifies files the agent changed (via `git diff` against initial commit)
   - Copies those files into `previous_solution/` in the current workspace
   - Commits `previous_solution/` so the agent can see it
4. The agent prompt mentions `previous_solution/` so it knows to reference prior work

**Key properties:**
- Optional: workspaces without `depends_on` work exactly as before
- Idempotent: if `previous_solution/` already exists, the copy is skipped
- Metadata files (workspace.yaml, .gitignore, etc.) are excluded from the copy

### Git-Based Reset

Each workspace is initialized as its own git repository with an initial commit
containing the starter code. This enables the Reset button to restore workspaces
to their original state using `git reset --hard`.

**Critical:** Workspaces MUST have their own `.git` directory. If missing, the
Reset button will throw an error: "Workspace is not a git repository".

**How it works:**
1. `bin/setup_workspace` copies course files and runs `git init`
2. Creates language-specific `.gitignore` (Python/Java/C)
3. Commits all files with message "Initial starter code"
4. Reset uses `git reset --hard <initial-commit>` + `git clean -fdx`

**Recreating workspaces:** If a workspace is missing `.git`, delete it and
recreate with `bin/setup_course <course> <agent> --force`.

### LLM Grading

LLM-based grading via OpenRouter supports two course types:

**Theoretical courses** (COMP 182, COMP 382): Written only, no code tests
**Hybrid courses** (COMP 140, COMP 321, COMP 322, COMP 341): Code tests + written component

The `submit` tool uses **detection-based grading**:
- If rubric exists in `{course}/rubrics/` AND writeup has content → LLM grade written
- If course has code tests → run them
- Combine scores for hybrid courses

**Rubric discovery convention** (checked in order):
- `{course}/rubrics/{assignmentName}_rubric.md`
- `{course}/rubrics/module{N}_rubric.md`
- `{course}/rubrics/hw{N}_rubric.md`

**Course-specific grading materials**:
- COMP 140: `rubrics/{module}_rubric.md` + `grading_prompt.md`
- COMP 182: `solutions/homework{N}_rubric.md` (combined rubric + solution)
- COMP 321: `rubrics/{project}_rubric.md` + `grading_prompt.md`
- COMP 322: `rubrics/hw{N}_rubric.md` + `grading_prompt.md`
- COMP 341: `rubrics/hw{N}_rubric.md` + `grading_prompt.md`
- COMP 382: `reference_solutions/hw{N}_solutions.md` + `grading_prompt.md`
- COMP 421: `solutions/{midterm,final}_rubric.md` + `grading_prompt.md`

**Output format** is compatible with existing test results, showing:
- Per-problem scores with rubric breakdown
- Detailed feedback for each problem
- Overall comments and patterns

### Code Review

LLM-based code review evaluates agent solutions across five dimensions (100 points total):
- Correctness & Completeness (20)
- Code Quality (20)
- Style & Conventions (20)
- Assignment Spirit (20)
- Robustness (20)

```bash
bin/bench-cli run assignment comp140_circles --model-key gpt-5-3-codex
bin/bench-cli phase llm-grade comp140_circles --model-key gpt-5-3-codex
bin/bench-cli phase code-review comp140_circles --model-key gpt-5-3-codex
```

**Multi-reviewer support:** Results saved as `code_review_result_<model>.md`. Run with different `--review-model` values to get multiple reviewer perspectives on the same code.

**Course-specific augmentation** (optional hooks, checked if present):
- `{course}/code_review_prompt.md` — additional review criteria
- `{course}/Shared/Coding_Style_and_Standards.md` or `{course}/style_guide.md` — style guide for adherence checks
- `{course}/reference_solutions/{name}_solution.{ext}` — reference implementation for comparison

## Agent Harness Architecture

The agent harness (`agent/`) provides a simple, model-agnostic way to benchmark AI agents:

```
┌─────────────────────────────────────────────────────────────┐
│                     bin/bench CLI                            │
│  (Runs agent on single-task workspace)                      │
└─────────────────────┬───────────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────────┐
│                    Agent Harness                             │
│  - Reads workspace.yaml for module info                     │
│  - Provides sandboxed tools (read, write, edit, grade)     │
│  - Uses AI SDK 6's ToolLoopAgent                            │
└─────────────────────┬───────────────────────────────────────┘
                      │
          ┌───────────┴───────────┐
          ▼                       ▼
┌─────────────────┐     ┌─────────────────────┐
│   OpenRouter    │     │   Sandboxed Tools   │
│   (300+ models) │     │   - read, write     │
│                 │     │   - edit            │
│                 │     │   - glob, grep      │
│                 │     │   - grade()         │
│                 │     │   - submit()        │
│                 │     │   - malloc_perf()   │
└─────────────────┘     └─────────────────────┘
```

**Key files:**
- `agent/index.ts` - Main exports (`runBenchmark`, `streamBenchmark`)
- `agent/tools.ts` - Workspace-sandboxed tools + `grade`, `submit`, and `malloc_perf` tools
- `agent/prompts.ts` - System prompts for benchmarking (Python, Java, C, theoretical)
- `agent/llmGrader.ts` - LLM-based grading for theoretical courses
- `agent/runner.ts` - High-level runner with logging

**Tool sandboxing:** All file operations restricted to workspace directory. Path traversal blocked.

**Two-phase malloc workflow:**
1. `grade` tool - Run correctness tests first
2. `malloc_perf` tool - Run performance benchmark after all tests pass

**Theoretical course workflow (comp182, comp382):**
1. Agent reads `instructions.md` to understand problems
2. Agent uses `write`/`edit` tools to write answers to `writeup.md`
3. Agent calls `submit` tool for final LLM-based grading
4. **One attempt only** - written submissions cannot be resubmitted

**Note:** Some comp182 assignments (HW4, HW6) are hybrid — they have both code tests
(run in Docker) and written components. These use `assignment_mode: "hybrid"` in their
config and follow the hybrid workflow instead.

are only available at `/clear/courses/comp421/pub/` on CLEAR. Net ID configurable via
`CLEAR_NETID` env var or `--netid` flag.

**Test workflow:** Upload source via SCP (tar bundle) -> compile on CLEAR -> run each test
(Lab 1: `xvfb-run ./<test>`; Lab 2: `yalnix yalnix <test>`; Lab 3: `mkyfs && yalnix yfs <test>`)
-> parse output (Lab 1: stdout; Lab 2/3: TTYLOG files) -> return structured results.

**LLM grading (Exams):** Midterm and final use LLM-based grading with rubrics in `solutions/`.
